# 대상별 보고서 작성 가이드

보고서의 효과는 독자에 따라 달라진다. 동일한 데이터라도 누가 읽느냐에 따라 톤, 깊이, 구조가 완전히 바뀌어야 한다. 이 가이드는 4가지 핵심 대상 유형별로 보고서를 최적화하는 방법을 정리한다.

---

## 1. 경영진 / 임원 (Executive)

### 톤 & 문체 가이드

**핵심 원칙:** 간결하고 결론 중심. 읽는 데 3분 이상 걸리면 실패다.

- 첫 문장에 결론 또는 핵심 제안을 배치한다
- 한 문단은 3문장 이내로 제한한다
- 수치와 비율로 근거를 제시한다 (정성적 표현 최소화)
- 능동태, 단정적 어미를 사용한다

**표현 예시:**

| 피해야 할 표현 | 권장 표현 |
|---|---|
| ~인 것으로 보입니다 | ~입니다 |
| 다양한 요인을 고려하면 | 핵심 요인 3가지는 다음과 같습니다 |
| 상당한 개선이 예상됩니다 | 15% 비용 절감이 예상됩니다 |
| 여러 부서에서 논의가 필요합니다 | 마케팅팀과 개발팀의 협의가 필요합니다 |
| 점진적으로 추진할 예정입니다 | Q2까지 1단계를 완료합니다 |

### 권장 분량

- **1~2페이지** (A4 기준)
- 본문: 500~800자
- 별첨으로 상세 데이터를 제공하되, 본문에서는 요약만 포함

### 중점 사항

- **의사결정 포인트:** "승인 / 보류 / 기각" 등 명확한 선택지를 제시
- **숫자 근거:** 매출, 비용, ROI, 시장 점유율 등 정량적 데이터 우선
- **리스크와 기회:** 핵심 리스크 1~2개와 그 대응 방안을 간략히 명시
- **타임라인:** 주요 마일스톤과 기한을 한눈에 파악할 수 있게 제시

### 피해야 할 것

- 기술적 세부사항이나 구현 방법론 서술
- 배경 설명에 절반 이상을 할애하는 구조
- "검토 부탁드립니다"로 끝나는 모호한 결론 (구체적 요청을 명시)
- 약어나 전문 용어를 설명 없이 사용 (경영진이 해당 도메인 전문가가 아닐 수 있음)
- 선택지 없이 상황만 나열하는 보고

### 추천 보고서 구조

```
1. 핵심 요약 (Executive Summary)
   - 결론 1문장
   - 핵심 수치 2~3개
2. 현황 (1~2단락)
3. 제안 사항 (선택지 + 각 장단점)
4. 기대 효과 (정량적)
5. 요청 사항 (승인, 예산, 인력 등 구체적 액션)
[별첨] 상세 데이터, 분석 근거
```

## 2. 팀 리더 / 중간관리자 (Team Lead / Middle Manager)

### 톤 & 문체 가이드

**핵심 원칙:** 구체적이고 실행 중심. "그래서 우리 팀은 뭘 해야 하는가"에 답해야 한다.

- 실행 항목(Action Item)을 명확히 구분하여 표기한다
- 담당자, 기한, 산출물을 구체적으로 명시한다
- 현실적 제약 조건(예산, 인력, 기술)을 솔직하게 기술한다
- 진행 상황을 추적할 수 있는 기준을 포함한다

**표현 예시:**

| 피해야 할 표현 | 권장 표현 |
|---|---|
| 빠른 시일 내에 완료 예정 | 3월 15일까지 완료 (담당: 김OO) |
| 관련 부서와 협력 필요 | 디자인팀 리뷰를 3/10까지 요청, 피드백 반영 후 3/14 최종본 확정 |
| 충분한 리소스 확보 후 진행 | 프론트엔드 개발자 2명 추가 배정 필요 (현재 1명 부족) |
| 전반적으로 양호한 진행 상태 | 전체 12개 태스크 중 8개 완료 (67%), 잔여 4개는 일정 내 완료 가능 |
| 리스크를 모니터링하겠습니다 | 리스크: API 응답 지연(발생 확률 중). 대응: 캐시 레이어 사전 구축(3/12 완료 목표) |

### 권장 분량

- **3~5페이지** (A4 기준)
- 본문: 1,500~3,000자
- 표, 간트 차트, 체크리스트 등 시각적 도구 적극 활용

### 중점 사항

- **실행 계획:** 단계별 태스크, 우선순위, 의존 관계를 명확히 정리
- **리소스 현황:** 투입 인력, 예산 사용 현황, 추가 필요 리소스
- **일정:** 마일스톤별 기한과 현재 진척률
- **이슈 & 리스크:** 현재 발생한 이슈와 잠재 리스크, 각각의 대응 방안
- **의사결정 요청:** 팀 리더 선에서 결정할 수 있는 사항과 상위 에스컬레이션이 필요한 사항 구분

### 피해야 할 것

- 지나치게 높은 수준의 요약만 제공 (실행에 필요한 구체성이 부족)
- 코드 레벨의 기술 상세 (팀원에게 위임할 수 있는 수준의 세부사항)
- 맥락 없는 숫자 나열 (수치에는 반드시 의미 해석을 덧붙임)
- 담당자나 기한 없는 액션 아이템
- 문제만 나열하고 대안을 제시하지 않는 보고

### 추천 보고서 구조

```
1. 요약 (프로젝트 상태 한눈에 보기)
   - 상태 표시: 정상 / 주의 / 위험
   - 핵심 지표 대시보드 (진척률, 예산 소진율 등)
2. 진행 현황
   - 완료 항목
   - 진행 중 항목 (진척률 포함)
   - 지연/블로커 항목
3. 이슈 & 리스크
   - 현재 이슈 (영향도, 대응 현황)
   - 잠재 리스크 (발생 확률, 대응 계획)
4. 실행 계획 (향후 1~2주)
   - 태스크 | 담당자 | 기한 | 산출물
5. 필요 사항
   - 의사결정 요청
   - 리소스 요청
   - 타 팀 협조 요청
[부록] 상세 일정표, 예산 내역
```

## 3. 동료 / 실무자 (Peer / Practitioner)

### 톤 & 문체 가이드

**핵심 원칙:** 상세하고 기술적. 동료가 이 보고서만 보고 같은 결과를 재현할 수 있어야 한다.

- 기술 용어를 자연스럽게 사용하되, 팀 외부에서 혼동될 수 있는 용어는 최초 사용 시 정의
- 코드 블록, 다이어그램, 스크린샷 등을 적극 활용한다
- "왜 이 방법을 선택했는가"에 대한 기술적 판단 근거를 포함한다
- 시도했으나 실패한 접근법도 기록하여 중복 시행착오를 방지한다

**표현 예시:**

| 피해야 할 표현 | 권장 표현 |
|---|---|
| 성능이 개선되었습니다 | p95 레이턴시가 120ms에서 45ms로 62.5% 감소 (N=10,000, 측정 기간: 7일) |
| 새로운 방식을 적용했습니다 | Redis 캐시 레이어를 도입하여 DB 직접 조회를 제거. 선택 근거: 읽기 비율 95%인 워크로드 특성 |
| 문제가 발생하여 수정했습니다 | 동시성 이슈(race condition)로 중복 주문 발생. 원인: 재고 차감 로직에 락(lock) 미적용. 해결: 분산 락(Redlock) 도입 |
| 잘 동작합니다 | 단위 테스트 148개, 통합 테스트 23개 통과. 커버리지 87%. 엣지 케이스 3건 추가 검증 완료 |

### 권장 분량

- **제한 없음** (필요한 만큼 상세하게)
- 단, 목차와 섹션 구분을 명확히 하여 필요한 부분만 빠르게 찾을 수 있게 구성
- 핵심 요약(TL;DR)을 문서 상단에 배치하여 전체를 읽지 않아도 핵심을 파악할 수 있게 함

### 중점 사항

- **기술 세부사항:** 아키텍처, 알고리즘, 데이터 모델, API 스펙 등
- **구현 방법:** 단계별 절차, 코드 예시, 설정 방법
- **의사결정 근거:** 대안 비교 분석 (A vs B vs C), 선택 이유
- **테스트 & 검증:** 테스트 전략, 결과, 커버리지, 엣지 케이스
- **재현 가능성:** 환경 설정, 의존성, 버전 정보 등

### 피해야 할 것

- 비즈니스 맥락 없이 기술 내용만 나열 (왜 이 작업이 필요한지 최소한의 배경은 제공)
- 검증되지 않은 주장 ("빠릅니다" 대신 벤치마크 수치 제시)
- 특정 개인의 로컬 환경에서만 재현 가능한 설명 (환경 독립적 기술)
- 오래된 정보를 업데이트 없이 그대로 유지
- 결론이나 권장사항 없이 데이터만 제시

### 추천 보고서 구조

```
1. TL;DR (3~5줄 핵심 요약)
2. 배경 & 문제 정의
   - 비즈니스 맥락 (간략히)
   - 기술적 문제 상세
3. 접근 방법
   - 검토한 대안들과 비교 분석
   - 선택한 방법과 그 근거
4. 구현 상세
   - 아키텍처 / 설계
   - 핵심 코드 또는 설정
   - 주의사항 및 트러블슈팅
5. 테스트 & 검증
   - 테스트 전략 및 결과
   - 성능 벤치마크
6. 향후 과제
   - 알려진 제한사항
   - 개선 방향
[부록] 전체 코드, 설정 파일, 참고 자료 링크
```

## 4. 외부 고객 / 파트너 (External Client / Partner)

### 톤 & 문체 가이드

**핵심 원칙:** 격식 있고 전문적. 우리 조직의 역량과 신뢰성을 보여주는 문서다.

- 존칭과 격식체를 일관되게 사용한다 (~합니다, ~드립니다)
- 내부 용어, 약어, 코드명을 사용하지 않는다
- 자사 브랜드 가이드라인에 맞는 표현을 사용한다
- 모든 수치와 주장에 출처 또는 근거를 명시한다

**표현 예시:**

| 피해야 할 표현 | 권장 표현 |
|---|---|
| 저희가 만든 시스템이 꽤 빠릅니다 | 본 솔루션은 평균 응답 시간 50ms 이내를 보장하며, 99.9%의 가용성을 제공합니다 |
| 이거 되게 잘 됩니다 | 도입 기업의 92%가 6개월 내 투자 대비 효과(ROI)를 달성하였습니다 |
| 우리 PJ에서 쓰던 걸 적용합니다 | 검증된 프레임워크를 기반으로 귀사의 환경에 최적화하여 적용합니다 |
| 문제 생기면 고쳐드립니다 | 전담 기술 지원팀이 24시간 이내에 대응하며, SLA에 따라 장애 복구를 보장합니다 |
| 싸게 해드리겠습니다 | 귀사의 규모와 요구사항에 맞춘 최적의 비용 구조를 제안드립니다 |

### 권장 분량

- **상황별 조정**
  - 제안서: 10~20페이지
  - 진행 보고: 3~5페이지
  - 결과 보고: 5~10페이지
- 시각적 완성도를 높이고, 표지/목차/페이지 번호 등 형식 요소를 갖춘다
- 요약본(1~2페이지)과 상세본을 분리 제공하는 것을 권장

### 중점 사항

- **가치 제안:** 고객이 얻게 될 구체적 비즈니스 가치 (비용 절감, 매출 증대, 리스크 감소 등)
- **신뢰성 근거:** 실적, 레퍼런스, 인증, 수상 이력 등
- **명확한 범위:** 포함/미포함 사항을 명확히 구분하여 기대치 관리
- **일정 & 비용:** 투명하고 현실적인 일정과 비용 구조
- **다음 단계:** 고객이 취해야 할 구체적 행동 (계약 서명, 킥오프 미팅 일정 등)

### 피해야 할 것

- 내부 프로세스나 조직 구조를 불필요하게 노출
- 경쟁사를 직접적으로 비하하는 표현
- 이행 불가능한 약속이나 과장된 수치
- 기술 구현 세부사항을 고객이 이해할 수 없는 수준으로 기술
- 비용이나 일정에 대한 모호한 표현 ("추후 협의" 남발)
- 오탈자, 서식 불일치 등 완성도가 떨어지는 문서 (신뢰도에 직접적 영향)

### 추천 보고서 구조

```
[표지] 제목, 날짜, 작성 조직, 보안 등급
[목차]

1. 개요 (Executive Summary)
   - 프로젝트/제안 배경
   - 핵심 가치 제안 (3개 이내)
2. 현황 분석
   - 고객 현재 상황 이해
   - 해결해야 할 과제
3. 제안 내용
   - 솔루션/서비스 개요
   - 적용 범위 (포함/미포함)
   - 기대 효과 (정량적)
4. 실행 계획
   - 단계별 로드맵
   - 주요 마일스톤
   - 투입 인력 및 역할
5. 비용 구조
   - 항목별 비용 내역
   - 투자 대비 효과(ROI) 분석
6. 레퍼런스
   - 유사 프로젝트 사례
   - 고객 추천사
7. 다음 단계
   - 구체적 후속 조치 안내

[부록] 상세 기술 사양, 약관, 회사 소개
```

## 대상별 빠른 비교표

| 항목 | 경영진/임원 | 팀 리더/중간관리자 | 동료/실무자 | 외부 고객/파트너 |
|---|---|---|---|---|
| **톤** | 간결, 단정적 | 구체적, 실행적 | 상세, 기술적 | 격식, 전문적 |
| **분량** | 1~2페이지 | 3~5페이지 | 제한 없음 | 상황별 |
| **첫 문장** | 결론/제안 | 프로젝트 상태 | TL;DR | 가치 제안 |
| **숫자 활용** | 핵심 KPI만 | 진척률, 예산 | 벤치마크, 커버리지 | ROI, 실적 |
| **기술 깊이** | 최소 | 중간 | 최대 | 고객 수준에 맞춤 |

## 활용 팁

1. **대상이 불명확할 때:** 작성 전에 "이 보고서를 읽고 무엇을 해야 하는 사람인가?"를 자문한다. 그 답이 대상 유형을 결정한다.

2. **복수 대상일 때:** 가장 높은 직급의 대상에 맞춰 본문을 작성하고, 상세 내용은 별첨으로 분리한다. 예를 들어, 경영진과 팀 리더가 모두 읽는 보고서라면 본문은 경영진 스타일로, 부록에 실행 계획을 첨부한다.

3. **구조 조합:** `structures.md`에서 정의된 보고서 구조와 이 가이드를 결합하여 사용한다. 대상 유형이 구조의 깊이와 포맷을 결정하고, 보고서 유형이 섹션 구성을 결정한다.

4. **톤 전환 연습:** 동일한 내용을 4가지 대상별로 다르게 쓰는 연습이 가장 효과적이다. 예를 들어, "서버 응답 시간이 느려졌다"를 각 대상에게 어떻게 전달할지 생각해 본다:
   - 경영진: "서비스 응답 지연으로 고객 이탈률이 3% 상승했습니다. 인프라 증설(예산 2억)을 요청드립니다."
   - 팀 리더: "API 응답 p95가 200ms를 초과. 원인: DB 커넥션 풀 부족. 3/15까지 커넥션 풀 확장 + 캐시 레이어 추가 예정 (담당: 박OO)."
   - 실무자: "RDS `max_connections` 150에서 병목 발생. pgbouncer 도입으로 커넥션 풀링 적용 예정. 설정값: `pool_mode=transaction`, `default_pool_size=50`."
   - 외부 고객: "일시적 서비스 지연이 발생하였으며, 현재 인프라 최적화를 진행 중입니다. 3월 중 완료 예정이며, 완료 후 응답 시간이 50% 이상 개선될 것으로 예상됩니다."

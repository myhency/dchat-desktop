# 보고서 구조 가이드

> Claude가 Phase 2(구조 결정)에서 참조하는 레퍼런스.
> 주제 유형, 대상, 목적을 기반으로 최적의 보고서 구조를 추천한다.

---

## 1. 현황 → 문제 → 목적 → 전개 → 기대효과

### 추천 상황

문제를 인식하고 해결책을 제안해야 할 때 사용한다. 기획서, 개선 보고서, 신규 사업 제안서 등 "왜 해야 하는가"를 설득해야 하는 문서에 적합하다.

### 섹션 구성

| 섹션 | 역할 |
|------|------|
| **현황** | 현재 상태를 객관적 데이터와 함께 기술. 독자가 배경을 이해하도록 한다. |
| **문제** | 현황에서 도출되는 핵심 문제점을 명확하게 정의한다. |
| **목적** | 본 보고서(또는 프로젝트)가 달성하고자 하는 구체적 목표를 제시한다. |
| **전개** | 목적 달성을 위한 실행 방안, 일정, 자원 계획을 상세 기술한다. |
| **기대효과** | 실행 시 예상되는 정량적/정성적 효과를 제시하여 의사결정을 유도한다. |

### 짧은 예시

> **현황**: 당사 고객센터 평균 응대 시간은 8분 32초로 업계 평균(5분) 대비 70% 초과 상태이다.
> **문제**: 응대 시간 지연으로 고객 만족도(CSAT)가 전년 대비 15% 하락하였으며, 이탈률이 증가하고 있다.
> **목적**: AI 기반 상담 보조 시스템을 도입하여 평균 응대 시간을 5분 이내로 단축한다.

### 자동 추천 조건

- 주제 유형: 개선 제안, 신규 도입 기획, 문제 해결
- 대상: 중간 관리자, 실무 의사결정권자
- 목적: 승인 획득, 예산 확보, 변화 설득

## 2. SCQA (Situation → Complication → Question → Answer)

### 추천 상황

경영진이나 고위 의사결정권자를 설득해야 할 때 사용한다. 전략 보고서, 경영 제안서 등 논리적 스토리라인이 중요한 문서에 적합하다. 맥킨지(McKinsey) 스타일의 전략 문서에서 널리 활용된다.

### 섹션 구성

| 섹션 | 역할 |
|------|------|
| **Situation (상황)** | 모두가 동의하는 현재 상황을 간결하게 서술한다. 공감대를 형성한다. |
| **Complication (복잡화)** | 상황에 변화나 위협 요소가 발생했음을 제시한다. 긴장감을 만든다. |
| **Question (질문)** | 복잡화로 인해 자연스럽게 떠오르는 핵심 질문을 명시한다. |
| **Answer (답변)** | 질문에 대한 구체적 답변(전략/방안)을 근거와 함께 제시한다. |

### 짧은 예시

> **Situation**: 국내 SaaS 시장은 연 25% 성장 중이며, 당사는 시장점유율 3위를 유지하고 있다.
> **Complication**: 글로벌 1위 업체가 한국 시장에 본격 진출을 선언하면서 가격 경쟁이 심화될 전망이다.
> **Question**: 당사가 시장 지위를 방어하면서 수익성을 유지하려면 어떤 전략을 취해야 하는가?

### 자동 추천 조건

- 주제 유형: 전략 수립, 시장 대응, 방향성 제안
- 대상: 경영진, C-Level, 이사회
- 목적: 전략적 의사결정, 방향 설득, 투자 유치

---

## 3. 결론 선행형 (피라미드 원칙, Pyramid Principle)

### 추천 상황

읽는 사람이 시간이 부족하거나 빠른 의사결정이 필요할 때 사용한다. 핵심 결론을 먼저 제시하고 근거를 뒤에 배치하는 구조로, 바쁜 임원 대상 보고서나 의사결정 요청 문서에 최적이다.

### 섹션 구성

| 섹션 | 역할 |
|------|------|
| **핵심 결론** | 보고서의 최종 결론 또는 권고 사항을 첫 문단에 제시한다. |
| **핵심 근거 (3가지 이내)** | 결론을 지지하는 주요 논거를 MECE 원칙에 따라 나열한다. |
| **근거별 상세 설명** | 각 근거에 대한 데이터, 분석, 사례 등 뒷받침 자료를 기술한다. |
| **실행 제안** | 결론에 따른 구체적 다음 단계(Next Steps)를 명시한다. |

### 짧은 예시

> **핵심 결론**: B2B 영업 조직을 현행 지역 기반에서 산업 기반으로 재편할 것을 제안한다.
> **근거 1**: 산업 특화 영업팀의 수주율이 지역 기반 대비 40% 높다(내부 파일럿 결과).
> **근거 2**: 주요 경쟁사 3곳 중 2곳이 이미 산업 기반 체제로 전환을 완료했다.

### 자동 추천 조건

- 주제 유형: 의사결정 요청, 권고, 요약 보고
- 대상: 바쁜 임원, 최종 의사결정권자
- 목적: 신속한 판단 유도, 승인 요청, 핵심 전달

## 4. As-Is / To-Be

### 추천 상황

현재 상태에서 목표 상태로의 변화를 보여줘야 할 때 사용한다. 시스템 도입, 프로세스 개선, 조직 변경 등 "변화 전후"를 대비하여 보여주는 문서에 적합하다.

### 섹션 구성

| 섹션 | 역할 |
|------|------|
| **As-Is (현재 상태)** | 현재 프로세스, 시스템, 조직의 상태를 구체적으로 기술한다. |
| **문제점/개선 필요사항** | As-Is에서 도출되는 비효율, 리스크, 한계를 정리한다. |
| **To-Be (목표 상태)** | 개선 후 달성할 이상적 상태를 명확하게 그린다. |
| **Gap 분석** | As-Is와 To-Be 사이의 차이를 항목별로 비교 분석한다. |
| **전환 계획** | Gap을 해소하기 위한 단계별 실행 계획과 일정을 제시한다. |

### 짧은 예시

> **As-Is**: 현재 재고 관리는 엑셀 기반 수작업으로 운영되며, 월 1회 수동 실사를 진행한다.
> **To-Be**: WMS(창고관리시스템) 도입으로 실시간 재고 추적 및 자동 발주 체계를 구축한다.
> **Gap**: 시스템 부재 → WMS 도입, 수동 실사 → 바코드/RFID 자동화, 월간 집계 → 실시간 대시보드.

### 자동 추천 조건

- 주제 유형: 시스템 도입, 프로세스 변경, 디지털 전환, 조직 개편
- 대상: IT 부서, 운영 관리자, 프로젝트 스폰서
- 목적: 변화 정당화, 투자 근거 제시, 이해관계자 합의

## 5. STAR (Situation → Task → Action → Result)

### 추천 상황

이미 수행한 일의 성과를 보고할 때 사용한다. 프로젝트 완료 보고, 성과 보고서, 우수 사례 발표 등 "무엇을 했고 어떤 결과를 얻었는가"를 정리하는 문서에 적합하다.

### 섹션 구성

| 섹션 | 역할 |
|------|------|
| **Situation (배경)** | 프로젝트나 업무가 시작된 배경과 맥락을 설명한다. |
| **Task (과제)** | 주어진 목표, 요구사항, 제약 조건을 명시한다. |
| **Action (실행)** | 목표 달성을 위해 실제로 수행한 핵심 활동을 기술한다. |
| **Result (결과)** | 정량적/정성적 성과를 데이터와 함께 제시한다. 교훈도 포함한다. |

### 짧은 예시

> **Situation**: 2025년 3분기, 모바일 앱 사용자 이탈률이 월 12%까지 상승하였다.
> **Task**: 4분기 내 이탈률을 8% 이하로 낮추고, 핵심 기능 재사용률을 높여야 했다.
> **Action**: UX 리서치 기반으로 온보딩 플로우를 재설계하고, 푸시 알림 개인화를 적용하였다.

### 자동 추천 조건

- 주제 유형: 성과 보고, 프로젝트 완료, 우수 사례, 업무 회고
- 대상: 상위 관리자, 프로젝트 이해관계자, 인사평가 위원회
- 목적: 성과 입증, 기여도 전달, 교훈 공유

---

## 6. 5W1H 기반

### 추천 상황

사안을 빠짐없이 파악하고 전달해야 할 때 사용한다. 현황 분석, 조사 보고서, 사건 보고, 실태 파악 등 "전체 그림을 객관적으로 정리"하는 문서에 적합하다.

### 섹션 구성

| 섹션 | 역할 |
|------|------|
| **What (무엇)** | 대상 사안, 이벤트, 현상이 무엇인지 정의한다. |
| **Why (왜)** | 해당 사안이 발생한 원인 또는 분석이 필요한 이유를 설명한다. |
| **Who (누가)** | 관련 이해관계자, 담당자, 영향 대상을 명시한다. |
| **When (언제)** | 시간적 범위, 일정, 마감 기한을 기술한다. |
| **Where (어디서)** | 물리적/논리적 범위, 적용 지역/부서를 특정한다. |
| **How (어떻게)** | 진행 방법, 절차, 도구, 예산 등 실행 수단을 제시한다. |

### 짧은 예시

> **What**: 2026년 1분기 고객 클레임이 전분기 대비 35% 증가한 현상에 대한 조사 보고.
> **Why**: 클레임 증가의 주요 원인은 신규 배송 파트너 전환 과정에서의 배송 지연(평균 +2.3일)으로 파악된다.
> **Who**: 물류팀, CS팀, 신규 배송 파트너(XX로지스틱스) 관련.

### 자동 추천 조건

- 주제 유형: 현황 분석, 실태 조사, 사건 보고, 원인 분석
- 대상: 실무 담당자, 팀 리더, 감사/컴플라이언스 부서
- 목적: 사실 파악, 전체 상황 전달, 후속 조치 기반 마련

---

## 자동 추천 매트릭스

Claude가 Phase 2에서 사용자의 입력(주제 유형, 대상, 목적)을 분석하여 최적 구조를 추천할 때 아래 매트릭스를 참조한다.

### 주제 유형별 1차 매핑

| 주제 유형 | 1순위 구조 | 2순위 구조 |
|-----------|-----------|-----------|
| 문제 해결 / 개선 제안 | 현황→문제→목적→전개→기대효과 | As-Is / To-Be |
| 전략 수립 / 방향 제안 | SCQA | 결론 선행형 |
| 시스템 도입 / 프로세스 변경 | As-Is / To-Be | 현황→문제→목적→전개→기대효과 |
| 의사결정 요청 / 승인 요청 | 결론 선행형 | SCQA |
| 성과 보고 / 프로젝트 완료 | STAR | 5W1H |
| 현황 분석 / 조사 보고 | 5W1H | STAR |
| 신규 사업 기획 | 현황→문제→목적→전개→기대효과 | SCQA |
| 사건/사고 보고 | 5W1H | 현황→문제→목적→전개→기대효과 |

### 대상별 가중치 조정

| 대상 | 선호 구조 | 이유 |
|------|-----------|------|
| 경영진 / C-Level | 결론 선행형, SCQA | 시간 부족, 핵심 우선 |
| 중간 관리자 | 현황→문제→목적→전개→기대효과 | 맥락과 실행 계획 모두 필요 |
| 실무 담당자 | 5W1H, As-Is / To-Be | 구체적 사실과 변화 내용 중심 |
| 외부 이해관계자 / 투자자 | SCQA, 결론 선행형 | 논리적 스토리와 핵심 결론 중심 |
| 인사평가 / 감사 위원회 | STAR, 5W1H | 성과 근거와 객관적 사실 중심 |

### 목적별 최종 확인

| 목적 | 최적 구조 |
|------|-----------|
| 승인/예산 획득 | 현황→문제→목적→전개→기대효과 |
| 전략적 설득 | SCQA |
| 빠른 의사결정 유도 | 결론 선행형 |
| 변화/전환 정당화 | As-Is / To-Be |
| 성과 입증 | STAR |
| 사실 전달/상황 파악 | 5W1H |

### 추천 알고리즘 (Claude 내부 로직)

```
1. 사용자 입력에서 주제 유형을 파악한다.
2. "주제 유형별 1차 매핑"에서 1순위 구조를 선택한다.
3. 보고 대상을 확인하고 "대상별 가중치 조정"을 적용한다.
   - 1차 매핑 결과와 대상 선호 구조가 일치하면 → 확정
   - 불일치하면 → 대상 선호 구조를 2순위로 제시
4. "목적별 최종 확인"으로 선택을 검증한다.
5. 최종 추천 시 1순위와 2순위를 함께 제시하고, 각각의 장단점을 간략히 설명한다.
```

### 복합 상황 처리

두 가지 이상의 구조가 동등하게 적합한 경우:

- **결합 사용**: 예를 들어 "시스템 도입 성과 보고"는 As-Is/To-Be로 변화를 보여주되, STAR로 실행 결과를 정리하는 하이브리드 구조를 제안한다.
- **분할 제안**: 하나의 보고서를 두 파트로 나누어 각기 다른 구조를 적용할 수 있음을 안내한다.
- **우선순위 기준**: 판단이 어려운 경우 "대상"을 최우선 기준으로 삼는다. 읽는 사람에게 맞추는 것이 가장 중요하다.
